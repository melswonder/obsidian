# 14章

マイクロサービス

  

  

　2010年代の中頃、マイクロサービスはソフトウェアエンジニアリング業界を席巻しました。マイクロサービスは、迅速な変更、スケーリング、そしてクラウドコンピューティングの分散的な特性への適合という、現代的な要求に対処することを意図した技術方式でした。多くの企業が、戦略的な決定として、巨大な一枚岩（モノリス）のコードベースを分解し、マイクロサービスが提供する柔軟性を手に入れようとしました。残念ながら、たくさんの失敗例が生まれました。多くの企業が手に入れたのは、柔軟性ではなく、分散した大きな泥団子、つまり分解しようとしたモノリスよりも、はるかに壊れやすく、不恰好で、高コストなシステムでした。

　歴史的には、マイクロサービスはドメイン駆動設計、特に区切られた文脈と関連づけられことがよくあります。ドメイン駆動設計の区切られた文脈とマイクロサービスを同義語のように使う人がたくさんいます。しかし、この二つは、本当に同じものなのでしょうか。この章では、ドメイン駆動設計という設計の方法論と、マイクロサービスという実装の方法論の関係を探求します。二つの方法論がお互いにどう影響しあうか、そして、重要な点として、マイクロサービスベースのシステムを効果的に設計するために、ドメイン駆動設計をどう活用できるかを学びます。

　まず、基礎知識として、サービスそしてマイクロサービスとは何であるかの定義から始めましょう。

14.1　サービスとは何か？

　OASIS（コンピュータと通信に関する標準化団体の一つ）によれば、サービスは「一つ以上の機能にアクセスできる仕組みで、規則として定められたインターフェースを用いて提供されるもの」と定義されています[＊1](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#footnote-009)。規則とした定められたインターフェースとは、サービスへのデータ入力、あるいは、サービスからのデータ出力の何らかの仕組みです。リクエスト/レスポンスモデルのような同期式のインターフェースもあれば、イベントに対するプロデューサー/コンシューマーモデルのような非同期式のインターフェースもあります。[図14‐1](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#i100)のように、サービスの公開インターフェースが他のシステムコンポーネントと通信し連係する手段を提供します。

14.2　マイクロサービスとは何か？

　マイクロサービスの定義は驚くほどシンプルです。サービスは公開インターフェースによって定義されます。ですから、マイクロサービスとは、きわめて小さな公開インターフェース、つまりきわめて小さな玄関ドアを持つサービスです。

　公開インターフェースを小さくすれば、サービスの機能が明快になり、他のコンポーネントとのつながり方も理解しやすくなります。また、サービスの機能が減ることで、それぞれのサービスの変更理由が限定的になり、開発、管理そしてスケーリングの独立性が高くなります。

　それに加えて、この特性は、マイクロサービスがデータベースを公開しない理由につながります。サービスの玄関ドアの一部としてデータベースを外部にさらしてしまえば、公開インターフェースは巨大化します。たとえば、データベースで実行可能なSQL文の組み合わせ方はどれくらいあるでしょうか。SQLはとても柔軟な言語です。組み合わせの数は、おそらく無限になるでしょう。だからこそ、マイクロサービスはデータベースをカプセル化して隠蔽し、データへのアクセス方法を、連係に特化したコンパクトな公開インターフェースに限定します。

14.2.1　「メソッド単位のサービス」が完璧なマイクロサービスか？

「マイクロサービスとは、きわめて小さな公開インターフェースである」という説明は、単純に見えます。この説明だと、サービスの公開インターフェースをたった一つのメソッドに限定することが、完璧なマイクロサービスになるように聞こえるかもしれません。このような、ばか正直な分割方法を実践した場合、何が起こるか見てみましょう。

図14-3　サービスの単純な分割

　マイクロサービスのお作法に従い、サービスごとにデータベースをカプセル化しています。どのサービスも他サービスのデータベースを直接アクセスすることはできません。他のサービスのデータにアクセスするには、公開インターフェースを使うことが必要です。しかし、現時点では、データをアクセスできる公開インターフェースはありません。しかし、サービスは、お互いに連係し、自サービスで行う変更を他サービスと同期することが必要です。連係して同期するためには、それぞれのサービスの公開インターフェースの拡張が必要です。この拡張の結果を可視化すれば、サービスどうしの連係とデータの流れは、典型的な「分散した大きな泥団子」になっています（[図14‐4](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#i103)）。

　ランディ・シャウプの比喩で言い換えれば、システムを非常に細かい粒度のサービスに分解することで、すべてのサービスの玄関ドアはきわめて小さくなりました。しかし、システム全体の機能を実現するには、各サービスに、スタッフ専用の裏口を大量に追加することになりました。この例から何を学べるでしょうか。

14.2.2　設計の目的

　一つのメソッドごとにサービスを分割する、という単純な分解方法が最善策でないことは明らかです。いろいろな理由があります。第一に、そもそも無理があります。サービスどうしを連係するために、連係用の公開インターフェースとそのための公開メソッドを拡張することを強いられました。第二に、個別の戦いには勝ちましたが戦争には負けました。個々のサービスの設計は当初のものよりもはるかに単純になりました。しかし、システム全体で見れば桁違いに複雑になってしまいました。

　マイクロサービスを採用する目的は、柔軟なシステムの構築です。単一のコンポーネントを設計することに注力し、システムを構成する他の部分との相互作用を無視することは、システムを簡潔に定義した以下のとらえ方に逆らっています。

- システムとは、一緒に動作するように接続されたモノや装置の集合である
- システムとは、特定の目的のために、一緒に使用されるコンピュータ機器とプログラムの集合である

　システムは孤立したコンポーネントの単なる寄せ集めではありません。マイクロサービスベースのシステムは、サービスがどんなに分割されていても、サービスどうしで連係し通信することが必要です。個々のマイクロサービスの複雑さとシステム全体の複雑さとの相互関係を見てみましょう。

14.2.3　システムの複雑さ

　40年前、クラウドコンピューティングは存在せず、グローバル規模の要求はなく、11.6秒[＊3](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#footnote-007)ごとにシステムをデプロイする必要もありませんでした。しかし、それでも、当時のエンジニアたちはシステムの複雑さと向き合っていました。現在とはツールが異なりますが、その時代に直面していた課題、そしてさらに重要なことに、その課題に対する解決策は、現代にも通じるものがあり、マイクロサービスベースのシステムを設計する時に、適用可能です。

複雑さを考える時、部分部分の局所的な複雑さの最小化よりも、はるかに重要な複雑さがあります。その複雑さとは、全体の複雑さ、つまりプログラム全体やシステム全体の構造です。言い換えれば、プログラムの主要部品どうしの関連性、あるいは相互依存性の度合いです。

　私たちの文脈にあてはめれば、「局所的な複雑さ」は個々のマイクロサービスの複雑さ、「全体の複雑さ」は複数のマイクロサービスで構成するシステム全体の複雑さに該当します。局所的な複雑さは、サービスの実装に依存し、全体の複雑さは、サービスどうしの相互作用と依存関係から決まります。マイクロサービスベースのシステムを設計する場合、どちらの複雑さを最適化することが重要でしょうか。両極端を分析してみましょう。

　全体の複雑さを最小にすることは意外と簡単です。コンポーネント間の相互作用を排除すればよいのです。つまり、すべての機能を一つのモノリシックなサービスとして実装してしまうやり方です。この戦略は特定のシナリオではうまくいくかもしれません。しかし、ほとんどの場合、恐ろしい「大きな泥団子」を生み出します。つまり、おそらく最悪の、局所的な複雑さです。

　一方、局所的な複雑さだけを最適化し、全体の複雑さを無視すると何が起こるかは、さきほどの例のとおりです。もっと恐ろしい「分散した大きな泥団子」の誕生です。この関係は[図14‐5](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#i104)のとおりです。

　マイクロサービスベースのシステムを適切に設計するには、全体の複雑さと局所的な複雑さの両方を最適化しなければなりません。どちらか一方だけの最適化を目標にした設計は、局所最適化です。両方の複雑さのバランスをとることが、全体最適化です。小さな公開インターフェースの考え方が、全体の複雑さと局所的な複雑さのバランスをとることに、どのように役立つのか見てみましょう。

14.2.4　マイクロサービスの深さ

　ソフトウェアシステム、あるいはあらゆるシステムのモジュールは機能とロジックの二つで定義されます。「機能」とは、そのモジュールが行うべきこと、つまり、何らかの業務機能です。「ロジック」は、業務機能を実現するための業務ロジックです。

　ジョン・オースターハウトは、著書『The Philosophy of Software Design』の中で、モジュラー性という考え方を検討し、モジュール設計の評価方法を、簡潔かつ強力な経験則として可視化しています。それは「深さ」による評価です。

　彼は、[図14‐6](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#i105)のような、長方形を使ってモジュラー性を可視化しています。長方形の上辺はモジュールの機能、 もしくはモジュールの公開インターフェースの複雑さを表現します。横に広がった長方形は提供する機能の広さを示し、横幅が狭いものは機能の限定、つまり簡潔な公開インターフェースを表します。長方形の大きさは、モジュールが実装する業務ロジックの量を表現しています。

![図14-6　深いモジュールと浅いモジュール](https://learning.oreilly.com/api/v2/epubs/urn:orm:book:9784814400737/files/image/i-105.jpg)

図14-6　深いモジュールと浅いモジュール

　このモデルで表現すると、効果的なモジュールは深くなります。つまり、簡潔な公開インターフェースが複雑なロジックをカプセル化します。効果的でないモジュールは浅くなります。つまり、公開インターフェースがカプセル化する複雑さは、深さのあるモジュールに比べて、かなり小さくなります。次のメソッドを考えてみましょう。

int AddTwoNumbers(int a, int b)   
{  
 return a + b;  
}

　これは浅いモジュールの極端な例です。公開インターフェース（メソッドのシグネチャ）の記述とロジック（メソッド）の記述は、ほとんどそのままです。このメソッドは、意味もなく、ただ動いているだけの部品です。このようなモジュールは、複雑さをカプセル化するどころか、システム全体に不必要な複雑さを持ち込みます。

14.2.5　深いモジュール

　用語の意味として、マイクロサービスが物理的な境界だけを表現するのに対し、深いモジュールは、論理的な境界と物理的な境界のどちらも表現します。この違いを除けば、両者の考え方の根底にある設計原則は同じです。

　[図14‐3](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#i102)に示した、メソッド単位に分割したサービスは、浅いモジュールです。サービスどうしで連係するための公開メソッドを導入する必要があり、結果として、インターフェースは本来あるべき姿よりもかなり広くなってしまいました。

　システムの複雑さの観点では、深いモジュールはシステムの全体的な複雑さを減少させます。一方、浅いモジュールは、局所的な複雑さをカプセル化しないコンポーネントを増やし、システム全体の複雑さを増加させます。

　また、浅いサービスは、マイクロサービスを目指したプロジェクトが失敗する原因です。「マイクロサービスとは、n行以下のコードで記述できるサービスである」、あるいは「修正するよりも書き換えるほうが簡単なサービスである」という誤った定義は、個々のサービスに気をとられすぎています。これは、設計のもっとも重要な側面、つまりシステム全体を見落とすことにつながります。

　システムをどこまでマイクロなサービスに分解するかのしきい値は、マイクロサービスで実現するユースケースから決めることが可能です。巨大な一枚岩を複数のサービスに分解すれば、変更時のコストは下がります。マイクロサービスまで分解すれば変更時のコストは最小化できます。しかし、適切なマイクロサービスのしきい値を越えて分解を続けると、深いサービスがどんどん浅くなります。インターフェースは再び拡大します。そうすると、連係の必要性から、変更コストが上昇します。そして、システム全体の構造は、恐ろしい分散した大きな泥団子になります。これを描いたものが[図14‐7](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#i106)になります。

　マイクロサービスとは何かを学びました。次に、深いサービスの境界を見つけるために、ドメイン駆動設計がどのように役立つかを見てみましょう。

14.3　ドメイン駆動設計とマイクロサービスの境界

　これまでの章で検討してきたドメイン駆動設計の手法のほとんどは、マイクロサービスと同様に、境界に関係します。つまり、区切られた文脈はモデルの境界であり、業務領域は業務活動の境界を区切り、集約と値オブジェクトはトランザクションの境界を表現しています。これらの境界のうち、どれがマイクロサービスの設計に役立つかを見てみましょう。

14.3.1　区切られた文脈

　マイクロサービスと区切られた文脈は、多くの共通点があり、しばしば同じ意味で使われます。しかし、本当にそうでしょうか。区切られた文脈の境界とマイクロサービスの境界は、相関しているのでしょうか。

　マイクロサービスも区切られた文脈も物理的な境界です。マイクロサービスは、区切られた文脈と同じように、単一のチームによって所有されます。そして、区切られた文脈と同様に、一つのマイクロサービスで複数のモデルは実装できません。なぜなら、インターフェースが不必要に複雑化するからです。マイクロサービスは確かに区切られた文脈です。しかし、逆の関係が成り立つのでしょうか。区切られた文脈はマイクロサービスであると言えるでしょうか。

14.3.2　集約

　区切られた文脈は、有効な境界をもっとも広い範囲で決定します。それに対して、集約はその逆を決定します。つまり、集約は、可能な範囲で、もっとも狭い境界を決定します。集約を、それ以上細かい単位の物理的なサービスや区切られた文脈に分解することは、次善の策にすらなりません。それどころか、[付録A](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-022.xhtml#t275)で学ぶように、控えめに言って、望ましくない結果になります。

　区切られた文脈と同様に、集約の境界はマイクロサービスの境界を決定する要素であると、しばしば考えられています。集約は、その内部に、業務ルール、不変条件そしてロジックをカプセル化し、業務機能としては、それ以上は分割できない単位です。しかし、この章の前半で学んだように、マイクロサービスを限界まで分解することは適切ではありません。マイクロサービスの単位は、他のコンポーネントとの相互作用の文脈でとらえる必要があります。つまり、以下の視点です。

- 対象とする集約は、同じ業務領域内の、他の集約とのやりとりがありますか？
- 他の集約と値オブジェクトを共有していますか？
- 集約の業務ロジックの変更が、業務領域の他のコンポーネントにどの程度影響を与えますか？あるいは、その逆に、他のコンポーネントの変更による影響をどの程度受けますか？

　集約という単位に分解した場合、同じ業務領域の他のオブジェクトとの関係が強いほど、単独のサービスとしては「浅く」なるでしょう。

　集約をサービスの単位とすることで、モジュール化された設計にできる場合もあります。しかし、多くの場合、このような細かい粒度のサービスはシステム全体の複雑さを増加させます。

14.3.3　業務領域

　マイクロサービスを設計する経験則として、区切られた境界や集約よりもバランスがとれているのは、サービスの境界を業務領域に合わせる方法です。[第1章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-002.xhtml#t007)で学んだように、業務領域は、細分化された事業活動であり、さまざまな業務活動が相互に連動します。業務領域は、企業が競合他社と競争するために必要となる、事業活動の構成要素です。事業活動の観点では、業務領域は事業が提供するサービス、つまり事業として何をするかを説明するものであり、サービスをどのように実現するかは説明しません。技術的な観点では、業務領域は強い関連性を持った一貫したユースケースの集合です。つまり、事業活動の同じモデルを使い、同じデータもしくは密接に関連するデータを処理し、機能的に強く関係します。[図14‐11](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#i110)に示すように、一つのユースケースの業務要件を変更すると、同じ業務領域の、他のユースケースに影響を与える可能性があります。

![図14-11　業務領域](https://learning.oreilly.com/api/v2/epubs/urn:orm:book:9784814400737/files/image/i-110.jpg)

図14-11　業務領域

　業務領域の粒度と提供する機能、つまり「どのように」ではなく「何を」に注目することで、業務領域は自然に深いモジュールとなります。業務領域が提供する機能は、実装の詳細である複雑な業務ロジックをカプセル化します。また、同じ業務領域に含まれるユースケースどうしの強い関係性は、結果としてモジュールの深さを保証します。多くの場合、ユースケースの集合を小さな単位に分割すると、公開インターフェースが複雑になり、モジュールとしては浅くなります。これらの理由から、業務領域に合わせてマイクロサービスの境界を設計することは、安全な選択肢です。

　マイクロサービスの境界を業務領域に合わせることが無難な経験則です。ほとんどのマイクロサービスにとってはそれが最適解です。とはいえ、業務領域以外の境界の区切り方をマイクロサービスの境界としたほうが適切な場合もあります。たとえば、区切られた文脈の同じ言葉の一貫性を重視して、広い範囲を境界とする場合、あるいは、非機能要件を満たすために集約をマイクロサービスの境界とする場合、などが該当します。マイクロサービスの設計判断は、事業の活動内容だけでなく、組織構造、事業戦略、非機能的要件にも依存します。[第11章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-014.xhtml#t155)で検討したように、ソフトウェアの全体的な構造と設計判断を環境の変化に常に適応させることが、きわめて重要です。

14.4　公開インターフェースを小さくする

　ドメイン駆動設計の考え方とやり方は、サービスの境界を見つけるだけでなく、深いサービスにすることにも役立ちます。この節では、共用サービスとモデル変換装置が、マイクロサービスの公開インターフェースをどのように簡素化できるかを説明します。

14.4.1　共用サービス

　共用サービスは、[図14‐12](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-018.xhtml#i111)に示すように、ある区切られた文脈のモデルと他のコンポーネントと連係するためのモデルを分離します。

　連係に特化したモデルと公開された言葉を導入することで、システム全体の複雑さが軽減します。第一に、利用する側に影響を与えることなくサービスの実装を改善できます。新たな実装モデルは既存の公開された言葉に変換可能です。第二に、公開された言葉は限定されたモデルを外部に公開します。公開された言葉は、連係の要件に合わせて設計されており、サービスの利用者には関係のない実装の複雑さをカプセル化して隠蔽します。たとえば、最小限のデータを使いやすいモデルとして利用者に公開可能です。

　ロジックの実装は同じでも、公開インターフェース（機能）を小さくすることで、サービスが「深く」なり、効果的なマイクロサービスとなります。

14.5　まとめ

　歴史的に見て、マイクロサービスの設計は、ドメイン駆動設計と深く結びついており、マイクロサービスと区切られた文脈という二つの用語は同義で使用されることが少なくありません。この章では、両者の関係性を分析し、二つが同じものではないことを確認しました。

　すべてのマイクロサービスは区切られた文脈です。しかし、すべての区切られた文脈が、必ずしも、マイクロサービスであるわけではありません。本質的には、マイクロサービスはサービスとしてもっとも小さい有効な境界を定義します。一方で、区切られた文脈は、それに含まれるモデルの一貫性を守りながら、もっとも広い有効なサービスの境界を示します。区切られた文脈の境界を広く定義すれば、巨大な泥団子になる可能性があります。一方、本来のマイクロサービスの境界よりも小さく分割すると、分散した大きな泥団子につながります。

　とはいえ、マイクロサービスとドメイン駆動設計とは密接な関係にあります。ドメイン駆動設計の考え方とやり方を使えば、効果的なマイクロサービスの境界を設計できることを確認しました。

　次の[第15章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-019.xhtml#t233)では、システムの全体構造について、別の観点から検討します。つまりイベント駆動型アーキテクチャによる非同期式の連係です。マイクロサービスの境界をさらに最適化するために、さまざまな種類のイベントメッセージを活用する方法を学びます。

    2. サービスの機能を実装するのに必要なコードの行数。この指標は行の幅に依存しないので、マイクロサービスの実装はウルトラワイドなモニターで行うことが望ましい
    3. マイクロサービスベースのシステムを設計する上で重要なマイクロサービスに適したミドルウェアや他のインフラストラクチャコンポーネントの調達。できれば、マイクロサービス分野で定評のあるベンダーから提供されるものが望ましい

    2. いいえ。集約は決して個別のマイクロサービスとして公開するべきではない
    3. 単一の集約からマイクロサービスを作ることは不可能である
    4. 対象領域によって判断が分かれる
