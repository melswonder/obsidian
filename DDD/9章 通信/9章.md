# 9章

通信

　この章で学ぶ内容は、以下のとおりです。

- 区切られた文脈どうしの通信方法
- 集約の設計原則に起因する連係方法の制約に対処する方法
- 複数の区切られた文脈にまたがる業務プロセスの実現方法

9.1　モデルの変換

　区切られた文脈はモデル、つまり同じ言葉の境界です。[第4章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-005.xhtml#t055)で学んだように、区切られた文脈どうしの連係方法は、いろいろな選択肢があります。異なる区切られた文脈を開発する二つのチームが、お互いの意図をうまく伝え、協力して開発を進めることに積極的ならば、良きパートナーの関係で二つの区切られた文脈を連係できるでしょう。連係の約束事（プロトコル）を必要に応じてうまく調整し、接続に問題が起きても、話し合いながらうまく対処できるでしょう。チームが緊密に協力するもう一つの方法としてモデルの共有があります。つまり、モデルの一部を共通部分として抽出し、進化させていきます。たとえば、区切られた文脈どうしを接続する部分を「契約」として抽出し、共有のソースコードリポジトリで管理します。

　利用者と供給者の関係に分類される連係方法では、二つのチームの力関係が、上流（サービスを提供する側）か下流（サービスを利用する側）のどちらかにかたよります。上流の区切られた文脈のモデルが、下流の区切られた文脈にとって、都合の悪い場合を考えてみましょう。この状況で上流と下流をつなぐためには、区切られた文脈のモデルを変換する、手の込んだ解決方法が必要です。

　モデルの変換は、どちらか一方で行うこともあれば、場合によっては両方で行うこともあります。上流の区切られた文脈が提供するモデルを変換して利用するために、下流の区切られた文脈がモデル変換装置を用意することがあるでしょう。あるいは、上流の区切られた文脈のモデルの変更が、下流の区切られた文脈の実装に影響しないように、上流の区切られた文脈が共用サービスを提供し、下流の区切られた文脈が公開された言葉を利用する方法も可能でしょう。モデルを変換するロジックは、モデル変換装置と共用サービスで、同じような実装になります。ですから、この章では、この二つの連係方法を特に区別しません。必要な箇所に限って、両者の違いに言及します。

　モデルの変換ロジックは、状態なし（ステートレス）あるいは状態あり（ステートフル）のどちらかになります。「状態なし」のモデル変換は、通信中に即時に実行されます。共用サービスではリクエストが到着した時に、モデル変換装置ではリクエストを送出する時に、モデルの変換を実行します。一方、「状態あり」のモデル変換は、データベースを使った、もっと複雑な仕組みになります。このようなモデル変換をどのように実装するかを見てみましょう。

9.1.1　状態なしのモデル変換

　状態なしのモデル変換は、変換機能（上流の共用サービスまたは下流のモデル変換装置）を提供する側が、プロキシ（代理）方式[＊1](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#footnote-017)で実現します。つまり、通信中のリクエストを横取りして、送信元のモデルを送信先のモデルに変換します（[図9‐1](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#i058)）。

図9-1　プロキシ（代理）方式のモデル変換

　プロキシの実装方法は区切られた文脈間の通信方式が同期か非同期かで異なります。

9.1.1.1　同期式

　同期式でモデルを変換する場合、区切られた文脈のコードベースの一部に変換ロジックを埋め込むのが、典型的なやり方です（[図9‐2](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#i059)）。共用サービスであれば、公開された言葉への変換はリクエストを受け取った時に行います。モデル変換装置であれば上流の区切られた文脈にリクエストを送る時に変換します。

![図9-2　同期通信](https://learning.oreilly.com/api/v2/epubs/urn:orm:book:9784814400737/files/image/i-059.jpg)

図9-2　同期通信

　場合によっては、APIゲートウェイ機能を提供する外部のサービスに変換ロジックをまかせるほうが、低コストで便利かもしれません。APIゲートウェイはオープンソースのKongやKrakenDなどで実現できます。あるいはクラウドベンダーが提供するマネージドサービス（AWS API　Gateway、Google Apigee、Azure API Management）を利用できます。

　共用サービスを提供する区切られた文脈（上流）から見れば、文脈固有のモデルと連係に最適化した公開された言葉の変換は、APIゲートウェイが担当します。さらに言えば、独立したAPIゲートウェイを使うことで、複数バージョンのAPIを提供しやすくなるでしょう（[図9‐3](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#i060)）。

![図9-3　異なるバージョンの公開された言葉を提供する](https://learning.oreilly.com/api/v2/epubs/urn:orm:book:9784814400737/files/image/i-060.jpg)

図9-3　異なるバージョンの公開された言葉を提供する

　APIゲートウェイを使って実現したモデル変換装置は、複数の区切られた文脈から利用できます。この場合、モデル変換装置は、連係目的に特化した独立した区切られた文脈として機能します（[図9‐4](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#i061)）。

![図9-4　モデル変換装置の共有](https://learning.oreilly.com/api/v2/epubs/urn:orm:book:9784814400737/files/image/i-061.jpg)

図9-4　モデル変換装置の共有

　他のコンポーネントに対してモデル変換機能を提供するこのような区切られた文脈を「モデル交換文脈」（interchange contexts）と呼ぶことがあります。

9.1.1.2　非同期式

　非同期式の通信（非同期メッセージング）でモデルを変換する場合は「メッセージプロキシ」を用意します。つまり、発信元の区切られた文脈から発行されたメッセージを、中間コンポーネント（プロキシ）が代理で受け取ります。この中間コンポーネントが必要なモデル変換を行い、変換済みのメッセージを本来の購読者に転送します（[図9‐5](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#i062)）。

![図9-5　非同期通信での変換](https://learning.oreilly.com/api/v2/epubs/urn:orm:book:9784814400737/files/image/i-062.jpg)

図9-5　非同期通信での変換

　この中間コンポーネントは、メッセージのモデル変換だけでなく、受信側の区切られた文脈には関係のないメッセージをフィルタリングすることで、通信のノイズを減らすことが可能です。

　共用サービスを非同期メッセージングで実現する場合、モデルの変換が不可欠です。内部モデルのオブジェクトをそのまま公開された言葉として外部にさらけ出してしまうのは、ありがちな、まちがった設計です。業務イベントをそのまま送信してしまうのも、よくあるまちがいです。そういうまちがった設計は、区切られた文脈の内部モデルを外部に露出させます。非同期のモデル変換は、ある文脈で発生した業務イベントを横取りして公開された言葉に変換可能です。そうすることで、区切られた文脈の実装の詳細を適切にカプセル化できます（[図9‐6](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#i063)）。

　また、メッセージを公開された言葉に変換することで、内部イベントと公開イベントを明確に区別できます。内部イベントは区切られた文脈の内部だけで使うイベントです。公開イベントは他の文脈と通信するためのイベントです。この内部イベントと公開イベントの区別は[第15章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-019.xhtml#t233)で改めて取り上げます。[第15章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-019.xhtml#t233)ではドメイン駆動設計とイベント駆動型アーキテクチャがどう関係するかを検討します。

9.1.2　状態ありのモデル変換

　さらに重要なモデル変換として、状態ありのモデル変換があります。たとえば、複数のデータを集めて一つにまとめたい場合、あるいは、さまざまな発生元からのデータを統合して単一のモデルに変換したい場合です。このような場合、具体的にどうすればよいか検討してみましょう。

9.1.2.1　到着データを一つにまとめる

　区切られた文脈の性能を最適化するために、到着する複数のリクエストを一括して処理することを考えてみましょう。この場合、同期あるいは非同期のどちらでも、データを一つにまとめる操作が必要になるでしょう。

9.2　集約どうしの連係

　[第6章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-008.xhtml#t084)で検討したように、集約が他のコンポーネントと連係するやり方として、業務イベントの発行（publish）があります。外部のコンポーネントは、必要に応じて業務イベントを購読（subscribe）し、業務ロジックを実行します。業務イベントの発行は、メッセージ通信基盤を使って実装します。具体的な実装方法を見てみましょう。

　その前に、業務イベントを発行する時にやりがちな誤りと、それが引き起こす結果について考えてみましょう。 次のコードを見てください。

01 public class Campaign  
02 {  
03   ...  
04   List<DomainEvent> _events;  
05   IMessageBus _messageBus;  
06   ...  
07   
08   public void Deactivate(string reason)  
09   {  
10     for (l in _locations.Values())  
11     {  
12       l.Deactivate();  
13     }  
14   
15     IsActive = false;  
16   
17     var newEvent = new CampaignDeactivated(_id, reason);  
18     _events.Append(newEvent);  
19     _messageBus.Publish(newEvent);  
20   }  
21 }

　17行目でイベントを作成しています。その後に続く2行で、そのイベントを集約内部の業務イベントのリストに追加し（18行目）、そのイベントをメッセージ通信基盤に発行しています（19行目）。業務イベントを発信するこの実装方法は、シンプルですが、まちがっています。集約の内部から業務イベントを発行するこのやり方がよくない理由は二つです。一つは、データベースに新しい状態をコミットする前に、業務イベントを外部に発信している点です。購読側はキャンペーンが停止されたという通知を受け取ります。しかし、その時点ではキャンペーンの状態と不整合になっている可能性があります。もう一つの問題は、データベースへのコミットが失敗した場合です。書き込みが競合した場合、集約内部の後続ロジックがその操作を無効と判定した場合、あるいは障害でデータベースへの書き込みが失敗した場合、結果はどうなるでしょうか。データベーストランザクションはロールバックされます。しかし購読者に届いた業務イベントを撤回することはできません。

　違うやり方を考えてみましょう。

01 public class ManagementAPI  
02 {  
03   ...  
04   private readonly IMessageBus _messageBus;  
05   private readonly ICampaignRepository _repository;  
06   ...  
07   public ExecutionResult DeactivateCampaign(CampaignId id, string reason)  
08   {  
09     try  
10     {  
11       var campaign = repository.Load(id);  
12       campaign.Deactivate(reason);  
13       _repository.CommitChanges(campaign);  
14   
15       var events = campaign.GetUnpublishedEvents();  
16       for (IDomainEvent e in events)  
17       {  
18         _messageBus.publish(e);  
19       }  
20       campaign.ClearUnpublishedEvents();  
21     }  
22     catch(Exception ex)  
23     {  
24       ...  
25     }  
26   }  
27 }

　このコードでは業務イベントを発行する責任をアプリケーション層に移動しています。11行目から13行目の、`Campaign`（キャンペーン）集約をリポジトリから取り出し、`Deactivate`（停止）コマンドを実行し、更新した状態をデータベースにコミットすることに成功した場合だけ、15行目から20行目で、新規の業務イベントをメッセージ通信基盤に発行します。このコードは信頼できるでしょうか。答えはノーです。信頼できません。

　この場合、ロジックを実行しているプロセスは、何らかの理由で業務イベントの発行に失敗する可能性があります。たとえば、メッセージ通信基盤が停止しているかもしれません。あるいは、このコードを実行しているサーバーが、データベーストランザクションをコミットした直後に、つまり業務イベントを発行するよりも前に停止すれば、状態の不整合が発生します。データベーストランザクションはコミットされているが、業務イベントを発行できていない状態です。

　こういう例外的な状況も、送信箱（Outbox）を使うとうまく対処できます。

9.2.1　送信箱

　送信箱（[図9‐11](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#i068)）と呼ばれる以下の方法を使えば、業務イベントを確実に発行できます。

- 更新された集約の状態と業務イベントの両方を一つのトランザクションとしてデータベースにコミットする
- 中継サービスは新規にコミットされた業務イベントを読み取る
- 中継サービスは読み取った業務イベントをメッセージ通信基盤に発行する
- メッセージ送信が成功した場合、中継サービスはデータベースの業務イベントに送信済の印をつける、あるいは送信済となったイベントを完全に削除する

9.2.1.1　未送信のイベントを読み取る

　業務イベントの発行を中継するやり方として、プル方式とプッシュ方式の二つがあります。

プル方式（ポーリングして発行）

中継サービスが、継続的にデータベースを検索し、未送信の業務イベントを見つけます。データベースを繰り返しポーリングする負荷を最小限に抑えるためには、適切なインデックス設定が必要です。

プッシュ方式（トランザクションログを追跡）

データベースに用意された機能をうまく使えば、業務イベントが追加された時に中継サービスを呼び出せます。たとえば、トランザクションログを追跡して、レコードの更新や追加を検知できるリレーショナルデータベースがあります。NoSQL系のデータベースでは、AWSの DynamoDB Streamsのように、データベースへの変更のコミットをイベントストリームとして監視できるものがあります。

　送信箱によるメッセージ送信は「少なくとも1回（at least once）」を保証します。この意味を正しく理解しておくことが重要です。つまり、送信箱は、メッセージ送信の直後に何らかの障害が発生し、データベースで「送信済」に変更することに失敗した場合、次の読み取りサイクルで、同じメッセージが二重に送信されるでしょう。

　それでは次に、ここで説明した信頼性の高い業務イベントの発行方法を活用することで、集約の設計原則に起因する制約を克服する方法を見てみましょう。

9.2.2　サーガ

　トランザクションの範囲を一つの集約に限定することは、集約の重要な設計原則です。この原則は、集約の境界を注意深く検討し、関連性の強い業務機能を一つの集約に閉じ込めることを保証します。しかし、この原則を超えて、複数の集約にまたがる業務プロセスを実装しなければならない場合もあります。

　広告配信の例で考えてみましょう。広告キャンペーンが開始された場合、広告用の素材を自動的に広告事業者（パブリッシャー）に送ることが必要です。広告配信を開始した通知を事業者から受け取ったら、キャンペーンの状態を`Published`（公開済）に変更します。広告事業者が広告の配信を拒否した場合は、キャンペーンの状態を`Rejected`（配信拒否）に変更します。

　この業務フローは二つの業務要素、つまりキャンペーンと広告事業者にまたがっています。この二つの要素を一つの集約の境界内部に同居させて実装するのは、明らかにやりすぎでしょう。なぜならば、キャンペーンと広告事業者は、業務的には明らかに異なる役割を持っています。それぞれの業務領域は別の区切られた文脈に所属しているかもしれません。別の方法として、この業務フローを「サーガ」で実装することが可能です。

　サーガ[＊3](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#footnote-015)とは、開始から終了まで長く続く業務プロセスです。「長く続く」という意味は、必ずしも時間の絶対的な長さではありません。秒単位で終わるサーガもあれば、年単位で続くサーガもあります。重要なのはトランザクションの視点です。つまり、多数のトランザクションで構成される業務プロセスがサーガです。トランザクション処理を行うのは集約だけではありません。業務イベントを発行したり、コマンドに応答したりする、あらゆるコンポーネントがトランザクション処理を行う可能性があります。サーガは、関連するコンポーネントが発信する業務イベントを検知し、それに続くコマンドを別のコンポーネントに向けて発信します。一連の処理がどこかで失敗した場合、サーガはシステムの一貫性を確保するために、失敗に対応する「補償アクション」を実行します。

図9-13　サーガ

　この配信プロセスを実現するサーガは次のようになります。

　サーガは、次の三つのイベントを検知します（図の点線）。

- 集約`Campaign`が発行する`CampaignActivated`（キャンペーン開始）イベント
- 区切られた文脈`AdPublishing`（広告配信）が発信する`PublishingConfirmed`（配信開始）イベントと`PublishingRejected`（配信拒否）イベント

　サーガは、次の三つのコマンドを実行します（図の実線）。

- 区切られた文脈`AdPublishing`に対して`SubmitAdvertisement`（広告提出）コマンド
- 集約`Campaign`に対して`TrackPublishingConfirmation`（配信開始の追跡）コマンドと`TrackPublishingRejection`（配信拒否の追跡）コマンド

　この例では`TrackPublishingRejection`（配信拒否の追跡）コマンドが補償アクションです。この補償アクションによって、対象とする広告キャンペーンが公開済として扱われないことを保証します。実装は次のようになります。

public class CampaignPublishingSaga  
{  
  private readonly ICampaignRepository _repository;  
  private readonly IPublishingServiceClient _publishingService;  
  ...  
  
  public void Process(CampaignActivated @event)  
  {  
    var campaign = _repository.Load(@event.CampaignId);  
    var advertisingMaterials = campaign.GenerateAdvertisingMaterials();  
    _publishingService.SubmitAdvertisement(@event.CampaignId,  
                       advertisingMaterials);  
  }  
  
  public void Process(PublishingConfirmed @event)  
  {  
    var campaign = _repository.Load(@event.CampaignId);  
    campaign.TrackPublishingConfirmation(@event.ConfirmationId);  
    _repository.CommitChanges(campaign);  
  }  
  
  public void Process(PublishingRejected @event)  
  {  
    var campaign = _repository.Load(@event.CampaignId);  
    campaign.TrackPublishingRejection(@event.RejectionReason);  
    _repository.CommitChanges(campaign);  
  }  
}

　このコードでは、関連するイベントを配送するために、メッセージ通信基盤を利用しています。そしてイベントを受け取った時、適切なコマンドを実行します。このサーガは比較的単純です。つまり状態を管理しません。状態を管理するもっと複雑なサーガが必要になることもあるでしょう。たとえば、それぞれの操作の実行状況を追跡し、操作が成功しなかった時にしかるべき補償アクションを起動する仕組みです。このような場合、イベント履歴式の集約を使ってサーガを実装し、受け取ったイベントと発行したコマンドの完全な履歴を永続化します。しかし、コマンドを実行するロジックそのものは、サーガの外部に配置し非同期で実行すべきです。業務イベントを送信箱で発行するのと同じ、次のやり方です。

public class CampaignPublishingSaga  
{  
  private readonly ICampaignRepository _repository;  
  private readonly IList<IDomainEvent> _events;  
  ...  
  
  public void Process(CampaignActivated activated)  
  {  
    var campaign = _repository.Load(activated.CampaignId);  
    var advertisingMaterials = campaign.GenerateAdvertisingMaterials();  
    var commandIssuedEvent = new CommandIssuedEvent(  
      target: Target.PublishingService,  
      command: new SubmitAdvertisementCommand(activated.CampaignId,  
      advertisingMaterials));  
      
    _events.Append(activated);  
    _events.Append(commandIssuedEvent);  
  }  
  
  public void Process(PublishingConfirmed confirmed)  
  {  
    var commandIssuedEvent = new CommandIssuedEvent(  
      target: Target.CampaignAggregate,  
      command: new TrackConfirmation(confirmed.CampaignId,  
                      confirmed.ConfirmationId));  
  
    _events.Append(confirmed);  
    _events.Append(commandIssuedEvent);  
  }  
  
  public void Process(PublishingRejected rejected)  
  {  
    var commandIssuedEvent = new CommandIssuedEvent(  
      target: Target.CampaignAggregate,  
      command: new TrackRejection(rejected.CampaignId,  
                    rejected.RejectionReason));  
  
    _events.Append(rejected);  
    _events.Append(commandIssuedEvent);  
  }  
}

　この例では、`CommandIssuedEvent`（コマンド発行イベント）のインスタンスごとに、送信箱を使った中継方式で、適切なエンドポイント上でコマンドを実行します。業務イベントを送信箱で発行するのと同じように、サーガの状態変更とコマンド実行を分離します。そうすることで、プロセスの途中で障害が起きても、確実なコマンド実行を保証します。

9.2.2.1　一貫性

　サーガは複数のコンポーネントにまたがるトランザクションの実行を調整し、すべてのコンポーネントの状態は結果的に整合します。また、サーガは最終的にはすべてのコマンドを実行しますが、個々のトランザクションの実行は分離されています。この分離は、集約のもう一つの設計原則、つまり以下の原則に関係します。

データが強く整合していると想定できるのは、集約の境界の内側だけである。集約の境界の外側にあるデータは結果的に整合する。

　サーガを適切に使う指針として、この原則を使ってください。集約境界の設計が不適切なことを補うために、サーガを使ってはいけません。同じ集約に所属させる業務処理は、強い一貫性が保証されたデータを必要とします。

　サーガは、別の実装方法である、プロセスマネージャーと混同されることがあります。この二つは実装のやり方は似ていますが、異なる方式です。次の節ではプロセスマネージャーの用途を検討し、サーガと何が違うかを説明します。

9.2.3　プロセスマネージャー

　サーガは単純で直線的な業務フローを扱います。厳密に言えば、サーガは業務イベントとコマンドを対応させる仕組みです。サーガの説明に使った実装例は、以下のように、業務イベントとコマンドを単純に対応させています。

- `CampaignActivated`（キャンペーン開始）イベントを`PublishingService`（広告配信サービス）の`SubmitAdvertisement`（広告の提出）コマンドに対応させる
- `PublishingConfirmed`（配信開始）イベントを`Campaign`（キャンペーン）集約の`TrackPublishingConfirmation`（配信開始の追跡）コマンドに対応させる
- `PublishingRejected`（配信拒否）イベントを`Campaign`（キャンペーン）集約の`TrackPublishingRejection`（配信拒否の追跡）コマンドに対応させる

　それに対し、プロセスマネージャー（[図9‐14](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#i071)）は業務ロジックが中心となる複雑な業務プロセスを実装する方法です。プロセスマネージャーとは、プロセスの状態を管理し、次に処理すべきステップを決定する中央処理装置（CPU）です[＊4](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#footnote-014)。

図9-14　プロセスマネージャー

　わかりやすい経験則として、もしサーガの実装でif-else文を使った処理の分岐が必要なら、それはおそらくプロセスマネージャーです。

　プロセスマネージャーとサーガのもう一つの違いはその起動方法です。サーガは特定イベントを検知した時に、暗黙的に起動されます。さきほどのサーガの実装例では、`CampaignActivated`（キャンペーン開始）イベントを検知するとサーガが起動します。一方、プロセスマネージャーの実行は、特定のソースイベントと関連づけることはできません。そうではなく、プロセスマネージャーは、複数の処理ステップで構成する、開始から終了までが一つにつながった業務プロセスです。したがって、プロセスマネージャーは明示的に起動して、プロセスを開始する必要があります。具体例として、出張のために航空券とホテルを手配する業務を考えてみましょう。

　まず、割安なフライトを選定します。次に、そのフライトで問題がないことを出張予定者が確認します。もし別のフライトを希望する場合、フライトの変更は上司の承認が必要です。航空券を手配できたら、会社指定のホテルの中から宿泊するホテルを予約します。どのホテルも予約できなかった場合、航空券をキャンセルします。

　この例では、特定の業務要素（社員、フライト、ホテルなど）が出張手配のプロセスを開始するわけではありません。手配業務という処理全体をプロセスマネージャーとして実装することが必要です（[図9‐15](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-011.xhtml#i072)）。

図9-15　出張手配プロセスマネージャー

　実装の観点では、プロセスマネージャーは、多くの場合、複数の集約の組み合わせです。状態ありの集約、あるいはイベント履歴式の集約のどちらでも実装可能です。ソースコードで見てみましょう。

public class BookingProcessManager  
{  
  private readonly IList<IDomainEvent> _events;  
  private BookingId _id;  
  private Destination _destination;     // 目的地  
  private TripDefinition _parameters;   // 出張条件  
  private EmployeeId _traveler;         // 出張予定者  
  private Route _route;                 // 経路  
  private IList<Route> _rejectedRoutes; // 拒否された経路  
  private IRoutingService _routing;  
  ...  
  
  // 出張手配プロセスの初期化  
  public void Initialize(Destination destination,  
              TripDefinition parameters,  
              EmployeeId traveler)  
  {  
    _destination = destination;  
    _parameters = parameters;  
    _traveler = traveler;  
    _route = _routing.Calculate(destination, parameters);  
  
    var routeGenerated = new RouteGeneratedEvent(  
      BookingId: _id,  
      Route: _route);  
  
    var commandIssuedEvent = new CommandIssuedEvent(  
      command: new RequestEmployeeApproval(_traveler, _route)  
    );  
  
    _events.Append(routeGenerated);  
    _events.Append(commandIssuedEvent);  
  }  
  
  // 経路確定済  
  public void Process(RouteConfirmed confirmed)  
  {  
    var commandIssuedEvent = new CommandIssuedEvent(  
      command: new BookFlights(_route, _parameters)  
    );  
  
    _events.Append(confirmed);  
    _events.Append(commandIssuedEvent);  
  }  
  
  // 経路却下済  
  public void Process(RouteRejected rejected)  
  {  
    var commandIssuedEvent = new CommandIssuedEvent(  
      command: new RequestRerouting(_traveler, _route)  
    );  
  
    _events.Append(rejected);  
    _events.Append(commandIssuedEvent);  
  }  
  
  // 経路変更確定済  
  public void Process(ReroutingConfirmed confirmed)  
  {  
    _rejectedRoutes.Append(route);  
    _route = _routing.CalculateAltRoute(destination,  
                      parameters, rejectedRoutes);  
    var routeGenerated = new RouteGeneratedEvent(  
      BookingId: _id,  
      Route: _route);  
      
    var commandIssuedEvent = new CommandIssuedEvent(  
      command: new RequestEmployeeApproval(_traveler, _route)  
    );  
  
    _events.Append(confirmed);  
    _events.Append(routeGenerated);  
    _events.Append(commandIssuedEvent);  
  }  
  
  // 航空券手配済  
  public void Process(FlightBooked booked)  
  {  
    var commandIssuedEvent = new CommandIssuedEvent(  
      command: new BookHotel(_destination, _parameters)  
    );  
    
    _events.Append(booked);  
    _events.Append(commandIssuedEvent);  
  }  
  
  ...  
}

　このプロセスマネージャーは、明示的なプロセスIDと、永続化する状態を持っています。さきほどのサーガの例と同じように、プロセスマネージャーは予約業務の進行を制御する業務イベントとして、 `RouteConfirmed（`経路確定）、`RouteRejected（`経路却下）、`ReroutingConfirmed（`別経路確定）、などを購読します。そして`CommandIssuedEvent`（コマンド発行済イベント）を生成します。生成されたイベントは、送信箱を使ってしかるべき通知先に中継され、受け取った側で実際のコマンドが実行されます。

9.3　まとめ

　この章ではシステムの構成要素（区切られた文脈）を連係するための通信方法を学びました。最初に、モデル変換装置や共用サービスを実装するためのモデル変換のやり方を検討しました。モデル変換を逐次実行するやり方、あるいは、状態の追跡が必要になる複雑なロジックに対応するモデル変換のやり方を見てきました。

　送信箱は、集約が発行する業務イベントを外部に確実に送信する方法です。送信箱を使うことで、他のプロセスに障害が発生しても、業務イベントが必ず送信されることを保証します。

　複数のシステムコンポーネント（区切られた文脈）にまたがる、単純な業務プロセスを実装する方法がサーガです。業務プロセスが複雑な場合の実装方法が、プロセスマネージャーです。どちらの実装方法も、土台となるのは業務イベントへの非同期な応答とコマンド発行の仕組みです。
