# 7章

時間軸でモデルを作る

  

  

　前の章で、複雑な業務ロジックの実装方法として、ドメインモデルを説明しました。ドメインモデルを組み立てる部品、ドメインモデルの目的、そしてドメインモデルの使いどころを学びました。この章ではイベント履歴式ドメインモデルを説明します。基本となる考え方はドメインモデルと同じです。イベント履歴式ドメインモデルも、複雑な業務ロジックと中核の業務領域を対象にします。さらに言えば、モデルを構成する基本部品も同じです。つまり、値オブジェクト、集約そして業務イベントです。

　二つが異なるのは、集約の状態を永続化する方法です。イベント履歴式ドメインモデルでは、イベントソーシングを使って集約の状態を管理します。つまり、集約の状態を永続化するのではなく、集約の状態が変化したことを業務イベントで表現し、業務イベントの履歴を「真実を語る唯一の情報源」（the source of truth）として永続化します。

　この章では、最初にイベントソーシングという考え方を紹介します。次に、イベントソーシングとドメインモデルを組み合わせた、イベント履歴式ドメインモデル（event-sourced domain model）について説明します。

　データ内容を見ればわかるように、これは販売管理システムの見込み客に関するデータです。見込み客ごとに、管理番号、姓名、レコードの作成日時と更新日時、電話番号、そして商談の進捗状況を表現しています。

　商談の進捗状況を分析すると、見込み客への商談活動のプロセスは次のように想定できそうです。

- 商談は新規からスタートする
- 見込み客への商談の結果は、断られた（終了）、後日連絡することになった（商談予定）、商談が成立した（支払待ち）のいずれかになる
- 支払が完了すると成約済になる。支払が行われなかった場合は支払なしになる

　テーブルのカラム構成と実際のデータを調べると、これだけの情報が手に入ります。そしてこのデータモデルは同じ言葉を使って作られたと想像できます。しかし欠落している情報があります。何が欠落しているかわかりますか。

　このテーブルが表現しているのは、見込み客の「現在」の状態です。しかし、どのような経緯で現在の状態に到達したのかがわかりません。成約済になるまでに、電話を何回かけたでしょうか。すんなりと買ってもらえたのでしょうか。それともかなりのやりとりを繰り返したのでしょうか。商談の履歴から判断して、この見込み客と商談を続ける価値があるでしょうか。それとも、この見込み客との商談は終わりにして、もっと有望な見込み客との商談にとりかかったほうがよいのでしょうか。 このテーブルのデータには、こういう問いに答えるための情報が欠けています。このデータからわかるのは商談の現在の状態だけです。

　このような問いかけは、商談プロセスを最適化したいという、事業活動の本質的な関心事から生まれます。商談の記録を分析し、事実にもとづいて商談プロセスを最適化することは、事業活動にとって、きわめて重要な取り組みです。不足しているこのような情報を集める方法の一つがイベントソーシングです。

　イベントソーシングは、時間軸の視点を取り入れたデータモデルです。イベントソーシングを取り入れたシステムでは、集約の現在の状態を記録するかわりに、集約のライフサイクルで発生するすべての変化を一連のイベントとして永続化します。

　この業務イベントの履歴を使えば、顧客の最新状態を簡単に投影できます。次のように、それぞれのイベントに対して単純な変換ロジックを順次実行すればよいのです。

//見込み客状態モデルの投影  
public class LeadStateModelProjection  
{  
  public long LeadId { get; private set; }  
  public string FirstName { get; private set; }  
  public string LastName { get; private set; }  
  public LeadStatus Status { get; private set; }  
  public PhoneNumber PhoneNumber { get; private set; }  
  public DateTime? FollowupOn { get; private set; }  
  public DateTime CreatedOn { get; private set; }  
  public DateTime UpdatedOn { get; private set; }  
  public int Version { get; private set; }  
  
  // 新規登録イベント  
  // @eventは、予約語のeventを変数名として使うためのC#の記法  
  public void Apply(LeadInitialized @event)  
  {  
    LeadId = @event.LeadId;  
    Status = LeadStatus.NEW_LEAD;  
    FirstName = @event.FirstName;  
    LastName = @event.LastName;  
    PhoneNumber = @event.PhoneNumber;  
    CreatedOn = @event.Timestamp;  
    UpdatedOn = @event.Timestamp;  
    Version = 0;  
  }  
  
  // 架電イベント  
  public void Apply(Contacted @event)  
  {  
      UpdatedOn = @event.Timestamp;  
      FollowupOn = null;  
      Version += 1;  
  }  
  
  // 商談予定設定イベント  
  public void Apply(FollowupSet @event)  
  {  
      UpdatedOn = @event.Timestamp;  
      FollowupOn = @event.FollowupOn;  
      Status = LeadStatus.FOLLOWUP_SET;  
      Version += 1;  
  }  
  
  // 連絡先変更イベント  
  public void Apply(ContactDetailsChanged @event)  
  {  
      FirstName = @event.FirstName;  
      LastName = @event.LastName;  
      PhoneNumber = @event.PhoneNumber;  
      UpdatedOn = @event.Timestamp;  
      Version += 1;  
  }  
  
  // 注文受領イベント  
  public void Apply(OrderSubmitted @event)  
  {  
      UpdatedOn = @event.Timestamp;  
      Status = LeadStatus.PENDING_PAYMENT;  
      Version += 1;  
  }  
  
  // 支払完了イベント  
  public void Apply(PaymentConfirmed @event)  
  {  
      UpdatedOn = @event.Timestamp;  
      Status = LeadStatus.CONVERTED;  
      Version += 1;  
  }  
}

　記録されているすべての業務イベントに対して、イベントの種類ごとに多重定義した、`Apply`メソッドを順番に実行します。その結果、[表7‐1](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-009.xhtml#h002)のレコード内容（管理番号、姓名、電話番号）を正しく再現できます。

　`Version`フィールドに注目してください。業務イベントを適用するたびにバージョン番号を増やしています。バージョン番号はエンティティの状態を変更した回数です。さらに言えば、業務イベントを部分的に適用することで「タイムトラベル」ができます。つまり、ある時点までのイベントを適用して、その時点の状態を投影できます。たとえば、バージョン5の状態を知りたければ、最初の五つのイベントだけ適用すればよいのです。

　このやり方で投影できるのは特定時点の状態だけではありません。いくつかのシナリオを考えてみましょう。

7.1.1　検索

　検索機能を考えてみましょう。見込み客の連絡先、つまり姓名と電話番号は変更可能です。ですから、ある担当者が連絡先を変更した時に、別の担当者はその変更に気がつかず、変更前の姓名や電話番号で検索するかもしれません。こういう変更履歴まで含めた情報も、次のやり方で簡単に投影できます。

// 見込み客検索モデルの投影  
public class LeadSearchModelProjection  
{  
  public long LeadId { get; private set; }  
  public HashSet<string> FirstNames { get; private set; }  
  public HashSet<string> LastNames { get; private set; }  
  public HashSet<PhoneNumber> PhoneNumbers { get; private set; }  
  public int Version { get; private set; }  
  
  // 新規登録イベント  
  public void Apply(LeadInitialized @event)  
  {  
    LeadId = @event.LeadId;  
    FirstNames = new HashSet<string>();  
    LastNames = new HashSet<string>();  
    PhoneNumbers = new HashSet<PhoneNumber>();  
  
    FirstNames.Add(@event.FirstName);  
    LastNames.Add(@event.LastName);  
    PhoneNumbers.Add(@event.PhoneNumber);  
  
    Version = 0;  
  }  
  
  // 連絡先変更イベント  
  public void Apply(ContactDetailsChanged @event)  
  {  
    FirstNames.Add(@event.FirstName);  
    LastNames.Add(@event.LastName);  
    PhoneNumbers.Add(@event.PhoneNumber);  
  
    Version += 1;  
  }  
  
  // 架電イベント  
  public void Apply(Contacted @event)  
  {  
    Version += 1;  
  }  
  
  // 商談予定設定イベント  
  public void Apply(FollowupSet @event)  
  {   
    Version += 1;  
  }    
  
  // 注文受領イベント  
  public void Apply(OrderSubmitted @event)  
  {  
    Version += 1;  
  }  
  
  // 支払完了イベント  
  public void Apply(PaymentConfirmed @event)  
  {  
    Version += 1;  
  }  
}

　この投影ロジックは、`LeadInitialized`イベントと`ContactDetailChanged`イベントを使って、それぞれのイベントで発生した連絡先情報を生成します。他のイベントは連絡先情報を変更しません。

　この投影ロジックを小林裕美さんのイベント履歴に適用すると、変更前の情報も含んだ以下の内容を投影できます。

LeadId: 12  
LastNames: ['小林']  
FirstNames: ['浩美', '裕美']  
PhoneNumbers: ['555-2951', '555-8101']  
Version: 6

7.1.2　分析

　ビジネス分析チームから、見込み客についてもっと分析に適した形式のデータがほしいと言われました。分析チームが知りたいのは、見込み客へのフォローアップ電話の回数です。そして成約までこぎつけたか、あるいは途中で終了したかの結果と突き合わせて、効果的な商談活動のやり方を見つけようとしています。分析チームが必要としているデータを、次の方法で投影してみましょう。

// 分析モデルの投影  
public class AnalysisModelProjection  
{  
  public long LeadId { get; private set; }  
  public int Followups { get; private set; }　// 商談予定設定回数  
  public LeadStatus Status { get; private set; }  
  public int Version { get; private set; }  
  
  // 新規登録イベント  
  public void Apply(LeadInitialized @event)  
  {  
    LeadId = @event.LeadId;  
    Followups = 0;  
    Status = LeadStatus.NEW_LEAD;  
    Version = 0;  
  }  
  
  // 架電イベント  
  public void Apply(Contacted @event)  
  {  
    Version += 1;  
  }  
  
  // 商談予定設定イベント  
  public void Apply(FollowupSet @event)  
  {  
    Status = LeadStatus.FOLLOWUP_SET;  
    Followups += 1;  
    Version += 1;  
  }  
  
  // 連絡先変更イベント  
  public void Apply(ContactDetailsChanged @event)  
  {  
    Version += 1;  
  }  
  
  // 注文受領イベント  
  public void Apply(OrderSubmitted @event)  
  {  
    Status = LeadStatus.PENDING_PAYMENT;  
    Version += 1;  
  }  
  
  // 支払完了イベント  
  public void Apply(PaymentConfirmed @event)  
  {  
    Status = LeadStatus.CONVERTED;  
    Version += 1;  
  }  
}

　商談予定設定イベントを適用するロジックで、予定した回数（`Followups`）を数えるようにしました。この投影ロジックを使えば、既存のイベント履歴から、以下の情報を生成できます。

LeadId: 12  
Followups: 1  
Status: 成約  
Version: 6

　ここまでのコード例は、検索に最適化したモデルや分析に最適化したモデルをメモリ上で投影しています。しかし、この機能を実際に使えるようにするには、投影したモデルをデータベースに永続化する必要があります。次の[第8章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-010.xhtml#t107)で、この具体的なやり方、つまりコマンド・クエリ責任分離（CQRS）を学びます。

7.1.3　真実を語る唯一の情報源

　イベントソーシングを実現するには、オブジェクトのすべての状態変更をイベントとして表現し、永続化することが必要です。永続化されたイベントが「真実を語る唯一の情報源」（source of truth）となります。つまりイベント（event）ソーシング（sourcing）です。このやり方は[図7‐1](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-009.xhtml#i037)のようになります。

7.1.4　イベントストア

　イベントストアは追記専用です。記録したイベントの変更や削除はできません[＊2](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-009.xhtml#footnote-026)。イベントソーシングを実現するために、イベントストアで最低限必要な機能は次の二つです。

- 特定のエンティティに関するすべてのイベントを取り出す

　コードで表現すると次のようになります。

interface IEventStore  
{  
  IEnumerable<Event> Fetch(Guid instanceId);  
  void Append(Guid instanceId, Event[] newEvents, int expectedVersion);  
}

　`Append`メソッドの第3引数`expectedVersion`は楽観的な排他制御を実現するために必要です。つまり、新たなイベントを追記する時に、バージョンを引数に指定することで、競合の発生を検知します。もし指定したバージョンがデータストア側の最新バージョンよりも古い、つまり別のプロセスがイベントを追加済みであれば、イベントストアは書き込みを失敗させ例外を送出します。

7.2　イベント履歴式ドメインモデル

　前の章で説明したドメインモデルでは、集約は最新の状態を表現し、特定の業務イベントを発信します。それに対し、イベント履歴式ドメインモデルでは、集約のライフサイクルを業務イベントでモデル化します。集約のすべての状態変更を業務イベントで表現することが必要です。

　イベント履歴式集約への操作は以下の手順で行います。

1. 業務イベント履歴を読み込む
2. 業務判断の対象となる、ある時点の状態を業務イベント履歴から再構築する
3. 集約のコマンド（業務ロジック）を実行し、その結果として、業務イベントを生成する
4. 新たに生成した業務イベントをイベントストアにコミットする

　[第6章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-008.xhtml#t084)で説明した`Ticket`集約をイベント履歴式集約で実装してみます。

　まずアプリケーションサービスで、この四つの手順を実装します。つまり、チケットに発生した一連のイベントを取得し、集約を再構築し、適切なコマンドを実行し、状態の変化をデータベースに永続化します。コードは次のようになります。

「イベント履歴式ドメインモデル」という  
用語にした理由

「イベントソーシング」ではなく「イベント履歴式ドメインモデル」という用語にした理由を説明させてください。状態の遷移をイベントで表現するイベントソーシングは、さまざまな状態管理に適用可能です。イベントソーシングの適用対象は、ドメインモデルの部品である集約の状態管理に限定されません。そのため、集約のライフサイクルの状態管理にイベントソーシングを使っていることを明示するためには、イベント履歴式ドメインモデルという長い用語のほうが好ましいと考えました。

  

7.2.1　利点

　従来のやり方では、集約の最新状態をデータベースに記録します。それに比べ、イベント履歴式ドメインモデルを選択した場合、集約の設計はたいへんになります。しかし、この方法は重要な利点があります。次のような場面でこのやり方を検討する価値があります。

タイムトラベル

業務イベントの履歴から、集約の最新状態を構築するのと同じ方法で、過去のあらゆる時点の状態を再現できます。別の言い方をすれば、集約のあらゆる過去の状態を、任意に再現できます。

過去の再現は、システムの使われ方の分析、システムが行った判定結果の検査、そして業務ロジックの最適化などに使えます。

また障害や不具合の調査にも活用できます。つまり、障害が発生した時点の集約の状態を正確に再現できます。

深い洞察

[第Ⅰ部](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-001.xhtml#t006)で、中核の業務領域を最適化することは、事業活動において戦略的に重要であることを見てきました。イベントソーシングによって、システムの状態と使われ方を深く洞察できるようになります。この章で学んできたように、イベントソーシングは、業務イベントの履歴という一つの情報源から、さまざまな状態を表現できます。つまり、状態を投影する新たなロジックをいつでも追加できるので、既存のイベント履歴から、業務活動について新たな洞察を手に入れることが可能です。

監査ログ

業務イベントの履歴は厳密な一貫性を持つ監査ログです。集約の状態を変化させたすべての業務活動の正確な記録です。事業領域によっては、このような監査ログの保存が法的に義務づけられています。イベントソーシングを採用すればこの要求に簡単に対応できます。

監査ログ方式は、金銭や金融取引を管理するシステムで威力を発揮します。どのような計算判断が行われ、どのように資金が移動したかを簡単に追跡できます。

進化した楽観的排他制御

従来の楽観的な排他制御の仕組みでは、読み込んだデータを変更して書き込む途中で他のプロセスが同じデータを上書きしたことを検知すると、例外が発生し、書き込みは失敗します。

イベントソーシングを採用すると、既存のイベントを読み込んでから新しいイベントを書き込むまでに起きることを正確に把握できます。イベントストアに同時に追加された業務イベントを取得し、イベントを追加する操作が衝突しているのか、あるいは、それぞれの業務イベントは無関係であり、そのまま操作を継続しても安全かどうかを業務視点で判定するロジックを組み込めます。

7.2.2　欠点

　今までの説明だけ見れば、イベント履歴式ドメインモデルは、業務ロジックを実装する究極のやり方であり、できるだけこの方法を使うべきだと思うかもしれません。もちろん、こういう単純な見方は「事業活動の視点で設計判断する」という原則から外れます。イベント履歴式ドメインモデルの以下の課題を検討してみましょう。

学習曲線

このやり方の明らかな欠点は、従来のデータ管理の方法とまったく違うやり方であることです。このやり方を正しく実装するにはチームの訓練が必要です。新しい考え方に慣れるまでには、そうとうな時間がかかります。チームがイベント履歴式のシステムを開発した経験がない限り、この学習曲線を考慮に入れなければなりません。

モデルの発展性

イベント履歴式ドメインモデルを進化させることは、たいへんかもしれません。イベントソーシングの厳密な定義では、イベントは不変（イミュータブル）です。しかし、イベントのデータ構造を変えたい場合、どうすればよいでしょうか。いちど定義したイベントのデータ構造を変えるプロセスは、テーブル設計の変更に比べ、はるかに難しい課題です。実際、グレッグ・ヤングはこのテーマだけで本を1冊書いています[＊3](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-009.xhtml#footnote-025)。

イベントソーシングの実装は、システムにさまざまな仕組みを持ち込むことになり、システム全体の設計が複雑になります。この点は、次の章でCQRSを検討する時に詳しく取り上げます。

7.3　よくある質問

　イベントソーシングを初めて知った技術者は、たいてい似たような疑問を持ちます。そういうよくある質問にここで回答しておきしょう。

7.3.1　性能

イベント履歴から状態を再構築するこのやり方は、性能面が気になります。イベントが増えると、性能がかなり悪くなりそうですが、この方法でうまくいくのでしょうか。

イベント履歴から状態を投影するには、それなりの処理能力は必要です。そして、集約のイベント履歴が大きくなるほど性能が劣化します。

状態投影が性能にどのくらい影響するかは、実際に計測してみることが重要です。つまり、数百あるいは数千のイベント履歴の処理がどのくらい性能に影響するかの測定です。その測定結果と集約の現実的な大きさ、つまり一つの集約に対して発生が予想されるイベント数、を比較して検討してください。

ほとんどのシステムでは、一つの集約のイベントが一万を超えれば性能に影響がでるでしょう。実際には、集約のイベント数が百を超えるようなシステムはほとんどないでしょう。

性能に影響がでるほど巨大なイベント履歴を扱う必要がある特殊な状況では、別のやり方としてスナップショットがあります。スナップショットの実現方法は次のとおりです（[図7‐2](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-009.xhtml#i038)）。

- 独立したプロセスがイベントストアを監視し、新たなイベントを検知したら、最新の状態を投影してキャッシュに書き込む
- 集約に対する操作を実行するために、メモリー上に最新状態を投影する場合、以下を行う
    - 投影済の状態（スナップショット）をキャッシュから取得する
    - スナップショットを作った時点より後に追加されたイベントをイベントストアから取得する
    - 取得したイベントをメモリー上でスナップショットに適用する

このやり方だと大量のデータが発生しそうです。スケールできますか。

7.3.2　データの削除

イベントストアは追記専用のデータベースですが、データの物理削除が必要になった場合、どうすればよいでしょうか。たとえば、GDPR（EU一般データ保護規則）[＊4](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-009.xhtml#footnote-024)に適合するためです。

この問題は、イベントソーシングの設計パターンの一つである、「忘れられる内容」（forgettable payload）で解決します。まず、イベントに含まれる、慎重に扱うことが必要なすべての情報を暗号化します。この暗号化のキーはイベントストアとは別のKey-Valueストア保存します。Keyは集約の識別番号です。Valueは暗号化のキーです。暗号化された情報を削除したい場合、Key-Valueストアに保存した暗号化のキーを削除します。その結果、イベントに含まれる保護すべき情報は復元できなくなります。

7.3.3　他のやり方ではだめですか？

ログをテキストファイルに書き出して、それを監査ログとして使う方法ではだめですか。

データを業務系データベースとログファイルの両方に書き込むやり方は、まちがいを起こしやすいやり方です。本質的に異なる二つの記録方法、つまりデータベースへの書き込みとファイルへの書き込みの、トランザクション管理の問題です。もし片方の書き込みが失敗したら、もう片方の書き込みをロールバックする必要がありますが、たとえば、データベーストランザクションが失敗しても、先行して書かれたログファイルの記録は削除されません。したがって、このようなログは一貫性がなく、結果的に不整合（eventually inconsistent）となります。

状態ありのやり方、つまり、状態の書き込みと履歴テーブルへの書き込みを一つのデータベーストランザクションで処理する方法ではだめですか。

技術的な観点からは、この方法は状態と履歴の一貫性を実現しています。しかし、このやり方もまちがいを起こしやすい方法です。たとえば、将来プログラムの変更が必要になった時、保守担当の技術者が履歴テーブルに反映することを忘れてしまえばどうなるでしょう。

もっというと、状態変更ありのテーブルを「真実を語る唯一の情報源」とした場合、脇役となる履歴テーブルの扱いは、いいかげんになりがちです。履歴テーブルで情報が欠落し不適切な形式の書き込みがされることを防ぐ方法はありません。

状態を記録するテーブルにトリガーを追加して、すべての項目のスナップショットを履歴テーブルに複製する方法ではだめですか。

トリガーによる履歴の書き込みは、前の方法の欠点を克服できます。つまり、履歴テーブルへの書き忘れは起きません。とはいえ、この方法は単なるデータの記録、つまり項目の変化内容が記録されるだけです。このやり方では業務の文脈が欠落します。つまり、その項目が変更された理由を記録できません。理由がわからなければ、新たなモデルを投影できる可能性は、きわめて小さくなります。

7.4　まとめ

　この章ではイベントソーシングを説明しました。そして、その応用として、ドメインモデルの集約を時間軸の視点でモデル化する方法を説明しました。

　イベント履歴式ドメインモデルでは、集約の状態変化を一連の業務イベントで表現します。従来の方法、つまり集約の変更後の最新状態をデータベースに永続化する方法とは大きく異なります。集約の最新状態は業務イベントの履歴から投影します。さらに、業務ロジックの実装方法をイベント履歴式モデルにすることで、目的に合わせたさまざまなモデルを柔軟に投影できます。

　このやり方は業務データから深い洞察を得るためにきわめて重要です。業務活動の分析と最適化、あるいは法律が定める監査記録などに活用できます。
