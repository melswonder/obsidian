# 8章

技術方式

  

  

　ここまで、業務ロジックのモデリングと実装のやり方を検討してきました。この章では、もっと広い範囲の実装方法を探求します。つまり、アプリケーション全体を組み立てるために、コンポーネントどうしの相互作用と依存関係をどのように構造化するか、その方法です。

8.1　業務ロジックと技術方式

　業務ロジックは何よりも重要です。しかし、ソフトウェアの唯一のコンポーネントではありません。機能要件や非機能要件を満たすために、コードベース全体ではもっと多くの責務が求められます。入力情報を収集し出力を提供するために、ユーザーと対話する必要があります。状態を永続化するためにさまざまな永続化の仕組みを利用します。外部システムや情報提供サービスとの連係も必要です。

　このように、コードベースが扱うべき関心事は多岐にわたります。そのため、業務ロジックはあちこちに拡散しがちです。つまり、業務ロジックの一部がユーザーインターフェースやデータベースに実装されたり、異なるコンポーネントに重複して実装されたりします。実装の関心事を適切に構造化できていないコードベースは、変更がやっかいで危険になります。業務ロジックを変更しなければならない時に、どの部分が変更の影響を受けるのかがわかりにくくなります。一見無関係に見える部分に、予期せぬ影響が及ぶ可能性があります。逆に、変更しなければならないコードを見逃してしまうこともあります。これらの問題はすべて、コードベースの保守費用を劇的に増大させます。

　アプリケーションを構築するための技術方式（アプリケーションアーキテクチャ）は、コードベースのさまざまな関心事をどう構造化するかの原則を導入し、それぞれの関心事の境界を明確にします。つまり、業務ロジックを、システムの入力、出力、その他の基盤コンポーネント群とどう結びつけるかを定義します。どの技術方式を選択するかによって、コンポーネントの相互作用、つまり、コンポーネントどうしがどんな知識を共有し、どのようにお互いを参照するかが変わります。

　コードベース全体を適切に構造化する方法、あるいは的確な技術方式を選択することは、短期的には業務ロジックの実装を助けるために、長期的には保守費用を軽減するために、きわめて重要です。アプリケーションを構築するための、主要な三つの技術方式、つまり、レイヤードアーキテクチャ、ポートとアダプター、そしてCQRSを探求し、どういう場合にどの技術方式を使えばよいかを検討してみましょう。

8.2　レイヤードアーキテクチャ

　レイヤードアーキテクチャはもっとも一般的な技術方式の一つです。コードベースを階層構造で整理し、それぞれのレイヤーごとに異なる関心事を取り扱います。つまり、ユーザーとの対話、業務ロジック、そしてデータの永続化です。[図8‐1](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-010.xhtml#i040)はその概念図です。

　しかし、最近のシステムでは、プレゼンテーション層の役割は広がっています。つまり、プログラムの振る舞いを起動する、同期または非同期の、すべての手段が対象となります。たとえば、以下のようなものです。

- グラフィカルユーザーインターフェース（GUI）
- コマンドラインインターフェース（CLI）
- 他システムとの連係のためのAPI
- メッセージ通信基盤が配信するイベントの購読
- 送出イベントを配信するためのメッセージトピック

　これらはすべて、システムが外部環境からのリクエストを受け取り、処理結果を伝えるためのものです。厳密に言えば、プレゼンテーション層はプログラムの公開インターフェースです。

8.2.3　データアクセス層

　データアクセス層は、永続化の仕組みと接続する機能を提供します。もともとは、データベースを指していましたが、プレゼンテーション層の場合と同様に、現代のシステムではデータアクセス層の責務は広がっています。

　第一に、NoSQLによる永続化方式の革新と急激な普及によって、一つのシステムで複数のデータベースを利用することが一般的になってきました。たとえば、ドキュメントストアを業務系データベースとして利用し、検索インデックスを使って動的クエリを実行し、インメモリーデータベースでパフォーマンスを最適化する、といった具合です。

　第二に、従来のデータベースだけが情報を格納する媒体ではありません。たとえば、クラウドベースのオブジェクトストレージ[＊2](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-010.xhtml#footnote-022)を使ってシステムのファイルを保存したり、メッセージ通信基盤を介してプログラム間の通信を制御したりできます[＊3](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-010.xhtml#footnote-021)。

8.2.5　レイヤードアーキテクチャの派生型

　レイヤードアーキテクチャは、追加のレイヤー、つまりサービス層によって拡張されることもあります。

8.2.5.1　サービス層

アプリケーションの境界をサービス層で定義する。サービス層は利用可能な操作を定め、各操作に対するレスポンスを取りまとめる。

---

[＊4](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-010.xhtml#footnote-020-backlink)　マーチン・ファウラー著『エンタープライズアプリケーションアーキテクチャパターン』（翔泳社、2005年）

　サービス層はプレゼンテーション層と業務ロジック層との間の仲介役として機能します。次のコードを見てください。

namespace MvcApplication.Controllers  
{  
  public class UserController: Controller  
  {  
    ...  
  
    [AcceptVerbs(HttpVerbs.Post)]  
    public ActionResult Create(ContactDetails contactDetails)  
    {  
      OperationResult result = null;  
  
      try  
      {  
        _db.StartTransaction();  
          
        var user = new User();  
        user.SetContactDetails(contactDetails)  
        user.Save();  
          
        _db.Commit();  
        result = OperationResult.Success;  
      } catch (Exception ex) {  
        _db.Rollback();  
        result = OperationResult.Exception(ex);  
      }  
  
      return View(result);  
    }  
  }  
}

　このコントローラクラスは、プレゼンテーション層に属し、新しいユーザーを作成するエンドポイントを公開しています。このエンドポイントは、アクティブレコードのUserオブジェクトを使用して、新しいインスタンスを作成し、それを保存します。さらに、エラーが発生した場合、適切にハンドリングしてレスポンスを返せるように、データベースのトランザクション制御を行います。

　下位の業務ロジック層とプレゼンテーション層をさらに切り離すために、[図8‐6](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-010.xhtml#i045)に示すように、さまざまなアクション（制御ロジック）をサービス層に移動できます。

図8-6　サービス層

　重要なことは、技術方式の文脈では、サービス層は論理的な境界です。物理的に独立したサービスではありません。

　サービス層は、業務ロジック層のファサードとして機能します。つまり、公開するメソッドのインターフェースを定義し、下位レイヤーで実装された振る舞いをカプセル化します。例を示します。

interface CampaignManagementService  
{  
 OperationResult CreateCampaign(CampaignDetails details);  
 OperationResult Publish(CampaignId id, PublishingSchedule schedule);  
 OperationResult Deactivate(CampaignId id);  
 OperationResult AddDisplayLocation(CampaignId id, DisplayLocation newLocation);   
 ...  
}

　すべてのメソッドは、システムの公開インターフェースに対応しています。しかし、プレゼンテーションに関連する実装の詳細は含みません。プレゼンテーション層の責務は、サービス層への必要な入力の提供と、呼び出し元への応答の伝達に限定されます。

　では、先の例をリファクタリングし、制御ロジックをサービス層に抽出してみましょう。

namespace ServiceLayer  
{  
  public class UserService  
  {  
    ...  
      
    public OperationResult Create(ContactDetails contactDetails)  
    {  
      OperationResult result = null;  
  
      try  
      {  
        _db.StartTransaction();  
          
        var user = new User();  
        user.SetContactDetails(contactDetails)  
        user.Save();  
  
        _db.Commit();  
        result = OperationResult.Success;  
      } catch (Exception ex) {  
        _db.Rollback();  
        result = OperationResult.Exception(ex);  
      }  
  
      return result;  
    }  
  
    ...  
  }  
}  
  
namespace MvcApplication.Controllers  
{  
  public class UserController: Controller  
  {  
    ...  
  
    [AcceptVerbs(HttpVerbs.Post)]  
    public ActionResult Create(ContactDetails contactDetails)  
    {  
      var result = _userService.Create(contactDetails);  
      return View(result);  
    }  
  }  
}

　明示的なサービス層を持つことには、次の利点があります。

- 同じサービス層を複数の公開インターフェース（たとえばGUIとAPI）に使用できます。アプリケーションロジックの重複がなくなります。
- 関連するメソッドを一つの場所にまとめることで、モジュール性が向上します。
- プレゼンテーション層と業務ロジック層をさらに分離できます。
- 業務機能のテストがしやすくなります。

　ただし、サービス層が常に必要というわけではありません。たとえば、業務ロジックがトランザクションスクリプトで実装されている場合、それ自体がすでにシステムの公開インターフェースを形成する一連のメソッドとなっているため、サービス層は必要ありません。この場合、サービス層のAPIは、トランザクションスクリプトの公開インターフェースを復唱するだけであり、複雑さの抽象化やカプセル化はしません。したがって、サービス層または業務ロジック層のいずれかがあれば十分です。

　一方で、業務ロジックがアクティブレコードのような外部の制御ロジックを必要とする場合は、サービス層が必要です。この場合、サービス層をトランザクションスクリプトで実装し、サービス層が操作するアクティブレコードを業務ロジック層に配置します。

8.2.5.2　用語解説

　レイヤードアーキテクチャに対して次のような用語が使用されることがあります。

- プレゼンテーション層＝ユーザーインターフェース層
- サービス層＝アプリケーション層
- 業務ロジック層＝ドメイン層＝モデル層
- データアクセス層＝インフラストラクチャ層

　混乱を避けるために、本書では元からある用語を使って説明します。そうはいっても、個人的には「ユーザーインターフェース層」と「インフラストラクチャ層」が好みです。この用語のほうが、今どきのシステムの責務を適切に反映しているからです。また、物理的に独立したサービスとの混同を避けるためには、アプリケーション層もよい用語だと思います。

8.2.6　レイヤードアーキテクチャをいつ使うか

　レイヤードアーキテクチャは、業務ロジック層とデータアクセス層に依存関係があるため、業務ロジックをトランザクションスクリプトまたはアクティブレコードで実装するシステムに適しています。

　しかし、この技術方式でドメインモデルの実装をするのは難しいでしょう。ドメインモデルで業務を表現するオブジェクト（集約と値オブジェクト）は、基盤となるインフラストラクチャから独立し、基盤に関する知識を持つべきではありません。レイヤードアーキテクチャはトップダウン式の依存関係のため、この要件を満たすためには、何らかの工夫が必要です。ドメインモデルをレイヤードアーキテクチャで実装することは可能です。しかし、次に説明するポートとアダプターのほうがはるかに適しています。

  

補足：レイヤーとティアの違い

　レイヤードアーキテクチャはN層（N-Tier）アーキテクチャとしばしば混同されます。そしてその逆もまた然りです。この二つの方式は似ていますが、レイヤー（Layer）とティア（Tier）は異なる概念です。つまり、レイヤーは論理的な境界です。ティアは物理的な境界です。レイヤードアーキテクチャのすべてのレイヤーは、同じライフサイクルで変化します。つまり、レイヤーの集合が単一のユニットとして実装され、進化し、デプロイされます。一方、ティアは独立してデプロイできるサービスやサーバー、またはシステムです。[図8‐7](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-010.xhtml#i046)のN層システムを見てください。

![図8-7　N層（N-Tier）システム](https://learning.oreilly.com/api/v2/epubs/urn:orm:book:9784814400737/files/image/i-046.jpg)

図8-7　N層（N-Tier）システム

　このシステムは、Webベースのシステムに関わる物理サービス間の連係を表しています。利用者はデスクトップコンピュータやモバイルデバイスで動作するブラウザを使用します。ブラウザはリバースプロキシとやりとりし、Webアプリケーションにリクエストを送ります。WebアプリケーションはWebサーバー上で実行され、データベースサーバーと通信します。これらのコンポーネントは、コンテナで同じ物理サーバー上で動作することもあれば、複数のサーバーに分散して配置されることもあります。しかし、各コンポーネントはそれぞれ独立したデプロイと管理ができるため、これらはレイヤーではなくティアです。

　一方で、レイヤーとは、Webアプリケーション内部の論理的な境界のことです。

8.3.4　ポートとアダプターの同類

　ポートとアダプターは、ヘキサゴナルアーキテクチャ、オニオンアーキテクチャ、クリーンアーキテクチャとしても知られています。これらは用語の違いはあるものの、基本的な設計原則やコンポーネント間の関係は、以下のように、同じです。

- アプリケーション層＝サービス層＝ユースケース層
- 業務ロジック層＝ドメイン層＝コア層

8.3.5　ポートとアダプターをいつ使うか

　ポートとアダプターは、業務ロジックを技術的な関心事から切り離せるため、ドメインモデルを用いた業務ロジックの実装に最適です。

8.4　コマンド・クエリ責任分離

　コマンド・クエリ責任分離（CQRS：Command-Query Responsibility Segregation）は、業務ロジックと実行基盤の関係を、ポートとアダプターと同じ原則にもとづいて組織化します。しかし、この二つは、データの管理方法が異なります。CQRSはシステムのデータを複数の永続化モデルで表現することを可能にします。

　なぜそのような解決策が必要なのか、そしてどのように実装するのかを見てみましょう。

8.4.1　目的別のモデリング

　多くの場合、事業活動を単一のモデルで表現して、システムのあらゆる要件に対応することは困難です。たとえば、[第16章](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-020.xhtml#t248)で詳しく説明しますが、業務系のオンライントランザクション処理（OLTP：On-Line Transaction Processing）と分析系のオンライン分析処理（OLAP：On-Line Analytical Processing）では、システムのデータに対して異なる表現が必要となることがあります。

　複数のモデルを扱うもう一つの理由は、複数言語が混在する永続化（polyglot persistence）の概念に関連します。完璧なデータベースはありません。また、グレッグ・ヤング[＊6](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-010.xhtml#footnote-018)が言うように、すべてのデータベースはそれぞれの欠点があります。つまり、それぞれのデータベースに適した方法で、スケール、一貫性、クエリモデルのバランスをとらなければなりません。完璧なデータベースを見つける別の方法が、複数言語が混在する永続化です。つまり、異なるデータ要件を満たすために、複数のデータベースを併用する方法です。たとえば、あるシステムでは、業務系データベースとしてドキュメントストアを、分析とレポート用にカラムストアを、安定した検索性能を実現するために検索エンジンを使用するかもしれません。

　最後に、CQRSはイベントソーシングと密接に関連します。もともと、CQRSの狙いは、イベント履歴式モデルの持つ制約、つまりにイベントを取り込んで状態を投影するのは一つの集約のインスタンスだけ、という制約に対処することでした。CQRSは、複数モデルの投影結果をデータベース上に実体化し、さまざまなクエリ方法を柔軟に提供します。

　とはいえ、この章では、CQRSとイベントソーシングは切り離して説明しています。なぜなら、業務ロジックの実装方法がイベント履歴式ドメインモデルでなくても、CQRSは利用可能だからです。

　CQRSが、複数のストレージの仕組みを使って、システムのデータをどのように異なるモデルで表現するかを見てみましょう。

8.4.2　実装

　名前が示すように、CQRSはモデルの責任を隔離（segregate）します。モデルは2種類です。つまりコマンド実行モデルと読み取りモデルです。

8.4.2.1　コマンド実行モデル

　CQRSはシステムの状態を変更する操作（コマンド）を実行するために、コマンド専用の単一モデルを割り当てます。このモデルは、業務ロジックの実装、ルールの検証、そして不変条件の強制に使用します。

　また、このコマンド実行モデルは、データの強力な一貫性を保証する唯一のモデル、つまり「真実を語る唯一の情報源」です。業務を表現するオブジェクトに対して、一貫性のある状態を読み取り、更新時には楽観的な排他制御を行う必要があります。

8.4.2.2　読み取りモデル（投影）

　CQRSは、ユーザーに表示したり他のシステムに情報を提供したりするために必要な複数のモデルを定義できます。

　読み取りモデルは投影結果のキャッシュです。投影結果は、永続的なデータベース、フラットファイル、インメモリキャッシュなどに存在します。CQRSを適切に実装することで、投影されたすべてのデータを消去し、ゼロから再生成することも可能です。また、この方式により、将来的に新たな投影ロジックを追加して当初は想定していなかったモデルを投影できるように、システムを拡張できます。

　読み取りモデルは読み取り専用です。システムのどの操作も、読み込んだモデルのデータを直接修正できません。

8.4.3　読み取りモデルの投影方法

　読み取りモデルを正しく動作させるには、システムはコマンド実行モデルからの変更を、すべての読み取りモデルに投影する必要があります。この概念を[図8‐12](https://learning.oreilly.com/library/view/domeinqu-dong-she-ji-wohazimeyou-sohutoueanoshi-zhuang-toshi-ye-zhan-lue-wojie-bitukerushi-jian-ji-fa/9784814400737/xhtml/p-010.xhtml#i051)に示します。

8.4.4　課題

　非同期投影は、スケーリングやパフォーマンスにおいて利点がある一方で、分散コンピューティングの難しさがあります。メッセージの順序がばらばらのまま処理したり、重複して処理したりすると、読み取りモデルのデータの一貫性を保証できません。

　非同期投影は、新しい投影の追加や既存の投影の再生成が難しくなります。

　これらの理由から、常に同期投影を実装し、オプションとしてその上に非同期投影を追加することをおすすめします。

8.4.5　モデルの隔離

　CQRSでは、モデルの責務は、コマンド側とクエリ側で異なります。コマンドが操作するのは強力な一貫性を持つコマンド実行モデルだけです。クエリは、永続化された状態（読み取りモデルやコマンド実行モデル）を変更できません。

　CQRSに関する一般的な誤解は、コマンドはデータを変更するだけで、データを取得するのは読み取りモデルだけである、というものです。言い換えれば、コマンドは実行結果としてデータを返さない、という解釈です。しかし、これは誤りです。このアプローチは、不必要な複雑さを生み出し、ユーザー体験を損ないます。

　コマンドは、成功したか失敗したかを呼び出し側に常に知らせる必要があります。失敗した場合、なぜ失敗したのでしょうか。たとえば、妥当性の検証結果が失敗だったり、技術的な問題が発生したりしたのでしょうか。呼び出し側は、コマンドを修正する方法を知ることが必要です。したがって、コマンドはデータを返すことが可能ですし、ほとんどの場合、返すべきです。たとえば、ユーザー画面にコマンドの実行結果を反映しなければならない場合などです。この場合、利用者は、自分の操作に対するフィードバックをすぐに受け取れるため、作業が容易になるだけでなく、返された値を次の作業に利用できるので、不要なデータのやりとりを取り除けます。

　唯一の制限として、コマンドが返すデータは強い一貫性のあるモデル（コマンド実行モデル）を元にすることが必要です。なぜなら、投影されたデータは、結果的に整合するとしても、ただちに更新されるとは限らないからです。

8.4.6　コマンド・クエリ責任分離をいつ使うか

　CQRSが有用なのは、同じデータをもとに複数のモデルを扱う必要がある場合です。それぞれのモデルの保存先は異なる種類のデータベースになる可能性があります。業務の観点では、CQRSは、ドメイン駆動設計の中核の価値である、課題ごとにもっとも効果的なモデルを使えるようにします。そしてモデルを継続的に改善する活動を支援します。技術的な観点では、CQRSは、異なる種類のデータベースのそれぞれの強みを組み合わせて活用できます。たとえば、コマンド実行モデルを格納するためにリレーショナルデータベースを使い、全文検索のために検索インデックスを使い、高速データ取得のために事前レンダリングされたフラットファイルを使う、というように、複数の永続化の仕組みを、信頼できるレベルで同期された状態で利用可能にします。

　また、CQRSはイベント履歴式ドメインモデルに適しています。イベント履歴式ドメインモデルは、集約の状態にもとづいてレコードをクエリすることはできません。しかしCQRSは読み取り用のデータベースに状態を投影することでそれを実現します。

8.6　まとめ

　レイヤードアーキテクチャは、技術的な関心事にもとづいて、ソースコードを分解します。この方式は、業務ロジックとデータアクセスの実装を結びつけるため、アクティブレコードベースのシステムに適しています。

　一方、ポートとアダプターは依存関係を逆転させます。つまり、業務ロジックを中心に置き、業務ロジックをすべての基盤コンポーネントから独立させます。このやり方は、ドメインモデルで実装した業務ロジックに適しています。

　CQRSは、同じデータを複数のモデルで表現します。イベント履歴式ドメインモデルにもとづくシステムでは必須です。それ以外にも、複数の永続化モデルを用いる必要があるシステムに利用できる技術方式です。

　次章では、別の観点から技術的な実現方式を検討します。つまり、システム全体を構成するコンポーネント間で、どのように信頼できる相互作用（通信）を実現するかです。
