# 正規化とは
正規化とはデータベースで保持するデータの冗長性を排除し、一貫性と効率性を保持するデータ形式

適当にテーブル作ると情報が複数テーブルに存在して無駄なデータ領域と面倒な更新処理を発生させてしまうため(冗長性)


正規系は5まであるが　普通は3正規系まで理解すれば十分

### 主キーのルール
・同じ値が二度と出てきてはいけない
・空のデータがあってはいけない
・一度決めたら後で変更しない値が望ましい


上記2つの問題をもとに解決するには、次にようにテーブルを分けう方法がる。

![[スクリーンショット 2026-04-14 21.37.18.png]]
「扶養者」テーブルについては子を持っている社員レコードに限定にして
主キーがNULLになるのを防ぐ、また、それだけでは子なし社員が死滅するため
社員テーブルで社員情報を管理するようにすれば、子の有無にかかわらず、社員全員の情報を保持することが可能になる。 また、社員の名前と子の名前を結びつけたい時も「社員ID」をキーに結合すればできる

### なぜ1つのセルに複数の値を入れてはダメなのか？
セルに複数の値を許せば、主機が各列の値を一意に決定する時ことができないから、これは主キーの定義に違反しています、一方、スカラ値だけのテーブルならば主キーは各列を一意に決めることができる、したがって、リレーショナルデータベースは、必ず第1正規系を満たすテーブルを作らなければいけないのです。

実はこれは、正規系全体を利化するための鍵にになる理念と結びついている。それを
**関数従属性**と呼びます　xの入力に対してyを出力するものが悪ったとして、xに対してyが1つに決まります。　これを「yはxに従属する」と表現します、リレーショナルデータベースでは、このXとYの関数を次のように表現します

|X| -> |Y|

これは「Xの値を決めればYの値が1つに決まる」と言う数学の関数と同じことを意味しています。

実は正規化とは、テーブルのすべての列が、関数従属性を満たすように整理していくことです、

 '社員ID' -> '社員名'

と言う関係従属は成立しています　生かし

'社員ID' -> '子'

という関数従属は不成立です、社員ID「000A」に対して、子は「達夫、加藤」と言う二人が該当してしまうから

# 第2正規系 部分関数従属

「社員」テーブルをサンプルに使う
![[スクリーンショット 2026-04-15 15.07.37.png]]

このテーブルは第1正規系を満たしていることを確認しておきましょう、このテーブルは、いずれのセルもスカラ値から構成されているので、問題なく第1正規系であると言えます。
しかし、このテーブルはまだ、第2正規系ではありません。
その理由はこのテーブルにおける関数従属が「完全」ではないからです。

このテーブルは主キーは　「社員コード、社員ID」です、したがって、他のすべての列はこのキーに所属するのですが、よく見ると、「会社名」列だけでは、主キーの一部である「会社コード」に従属しています。　つまり以下のような関数従属性がある

	'会社コード' -> '会社名'

このように、主キーの1部分の列に対して、従属する列がある場合、この関係性のを　部分関数従属　と呼びます。これに対して、主キーを構成するすべての列に従属性がある場合を、　完全関数従属　と呼びます。

第2正規系とは、テーブル内で部分関数従属を解消し、完全関数従属のみのテーブルを作ることです

### 第2正規化を行う

部分関数従属を解消する手段も決まってます、「テーブルの分割」です

![[スクリーンショット 2026-04-15 15.56.23.png]]

　これによって、「社員」テーブルも新しく作った。「会社」テーブルも、すべての列が主キーに完全関数従属することになりました。「社員」テーブルは、もともと「会社名」列以外は完全関数従属していましたし、「会社」テーブルも、もともと完全関数従属の関係にあった、列の組み合わせでテーブルを作っているわけだから、完全関数従属するのは当然です。非常にシンプルな解決と言える

第2正規系にすることのメリットは、なんなのでしょう？

まず正規化する前の「社員」テーブルから考えましょう
![[スクリーンショット 2026-04-15 15.07.37.png]]
このテーブルを使って実際の業務を運用しようとすると困ったことが起こります

今の設計では、主キーが **{会社コード + 社員ID}** のセットになっています。データベースのルール上、主キーに NULL（空っぽ）は許されません。

- **困ること:** 「新しく取引を始めたC建設を登録したい」と思っても、まだそこに所属する社員が一人もいない場合、主キーの「社員ID」を埋めることができず、**会社そのものを登録できません。**
    
- **無理やりな回避策:** 「社員ID：999（仮）」のようなダミーデータを入れることになりますが、これはデータの正確性を損なう「汚い設計」の始まりです。

会社名は「会社コード」が決まれば自動的に決まるはずの情報なのに、すべての行に「会社名」を書き込む欄があります。

- **困ること:** 誰かが「C0001」の会社名を「A商事」ではなく「A商社」と打ち間違えて登録しても、データベースはそれをエラーにしてくれません。
    
- **「お手上げ」の理由:** * アプリの画面上ならチェック機能を付けられますが、エンジニアがコンソールから直接 SQL で `UPDATE` をかけたり、別のシステムからデータを流し込んだりすると、チェックをすり抜けて **「同じ会社コードなのに名前が違う」という矛盾したデータ** が完成してしまいます。

### 推移的関数従属
今度は「部署コード」と「部署名」のペアに注目します、今、「会社コード」が「C0001,A商事」について考えてみると「社員」テーブルからは、開発、人事、営業という3つの部署が存在することがわかります。しかしA商事の部署が3つだけである保証はありません。もしかしたら他の部署が残っているかもしれません。
こういう社員が1人もいない部署を、現在の第2正規化した状態の「社員」テーブルには登録できません (NULLのままレコードを登録できないため)

下記の二つの列間には

'部署コード'　-> '部署名' 

という関数従属が成立します。
一方社員IDと部署コードの間にも、当然のことながら

'会社コード,社員ID' -> '部署コード' 

という関数従属が存在しています。つまり、全体としては、

'会社コード,社員ID' -> '部署コード'  -> '部署名'

という二段階の関数従属があるわけです。このように、テーブル内部に存在する段階的な従属関数のことを、　推移的関数従属と呼びます


# 第3正規化を行う
この推移的関数従属によるデータ登録時の不具合を解消するには、第2正規系と同じように、テーブルを分割することで、それぞれの関数従属の関係を独立させます

![[スクリーンショット 2026-04-15 17.54.03.png]]

このように、新しく部署を管理するための「部署」テーブルを独立させることで、すべてのテーブルについて、非キー列はキー列に対してのみ従属するようになり、推移的関数従属も無くなりました。それにより、先ほど問題だった、一人もいない部署を登録できない。　という問題は解消されます「部署」テーブルに登録すると気には、社員情報が必要ないからです

# ボイスーゴット　世紀系
一般的な業務を設計する場合、ほとんどのケースで第３世紀系までを意識しておけば事足ります。　

世紀系において、3次のすぐ上位に位置するのがボイスーゴット世紀系です。これは考案者の二人の名前から取られており、BCNFという略称でも呼ばれています
「第３の次は第4世紀系では？」と思った人もいるが、間にボイスーゴットが定義されている

第3世紀系を満たしていて、ボイスーゴット世紀系を満たしていないテーブルはあまりお目にかからないのですが、例えば以下のようなテーブルです。

![[スクリーンショット 2026-04-15 20.13.46.png]]
このテーブルの意味は次のとおりです。テーブルの主キーは、「社員ID,チームコード」です。　　以下のような主キーから非キーへの関数従属性があります。

'社員ID,チームコード' -> 'チーム補佐'

一方、このテーブルにはもう一つの業務ルールが存在すると仮定します。
それは「チーム補佐」から「チームコード」に対する関数従属性がある、とするという過程です

'チーム補佐' -> 'チームコード'

このテーブルには、部分関数従属性および、推移的関数属性はありません。したがって、文句なく第３世紀系です。しかし、ボイスーゴット世紀系ではありません。　というのもボイスーごっと世紀系とは、「社員　チーム　リーダー」テーブルに見られるような非キーからキーへの関数従属をなくした状態を指すからです
